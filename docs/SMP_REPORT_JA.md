# Raspberry Pi 4 Xinu — 4コア(SMP)化レポート

**日付**: 2026-06-18  **対象**: `xinu-rpi4`(BCM2711 / 4× Cortex-A72, AArch64)
**カーネル**: `kernel8.img` md5 `8a3effa1`(全機能 + 4コア化を内包)

---

## 1. 結論(先に結果)

BCM2711 の **4つの Cortex-A72 コアのうち、これまで使われていたのはコア0の1個だけ**だった。残り3コア(コア1-3)を計算ワーカーとして起こし、並列計算を実行できるようにした。

実機(192.168.3.100)で素数カウントを **1コア vs 4コア**で計測した結果:

| n(上限) | 素数の数 | 1コア | 4コア | 速度向上 |
|---------:|---------:|------:|------:|:--------:|
|   300,000 |   25,997 |    53 ms |    17 ms | **3.11×** |
| 1,000,000 |   78,498 |   875 ms |   293 ms | **2.98×** |
| 2,000,000 |  148,933 | 2,383 ms |   802 ms | **2.97×** |
| 4,000,000 |  283,146 | 6,570 ms | 2,223 ms | **2.95×** |

- **`cores_online = 4`** … 4コア全てが起動・参加していることを実機で確認。
- **素数の数は 1コア時と 4コア時で完全一致**(correctness check passed)。値も π(n) の正しい値(π(10⁶)=78498 等)。
- 安定して **約3倍**の高速化。4倍に届かない理由は §6 で分析(SMP 機構のオーバーヘッドではなく、ワークロード由来の負荷不均衡)。

再現コマンド:
```
curl 'http://192.168.3.100/smp-bench?n=1000000'
```

## 2. なぜ「ワーカープール型」アーキテクチャにしたか(設計判断)

最初に2つのアーキテクチャを検討した。

| | A: ワーカープール型(採用) | B: 完全対称 SMP |
|---|---|---|
| コア0 | 現行 OS をそのまま実行 | 共有レディリストから実行 |
| コア1-3 | 専用計算ワーカー | 同上(対称) |
| レディリスト | コア0専有(無改造) | 全コア共有 + ロック |
| 必要なもの | volatile フラグ + barrier | IPI / per-core scheduler / spinlock |
| リスク | 低(OSは無改造) | **高** |

**B(完全対称 SMP)を採らなかった理由**:
このカーネルの **AIPL / actor / network(GENET)/ USB(xHCI)/ video(framebuffer)/ vheap スタックは、すべて「単一コア・協調実行・非再入」を前提に書かれている**(`system/proc.c` のコメントに、actor pump レース・vheap ロック・preempt 抑制の対策が多数)。4コアが対称にこれらを叩くと、これら全部を再入可能に書き換えるまで OS が不安定化する。実機ブリングアップでは「箱ごとハング → 物理電源再投入」のリスクが非常に高い。

**A を採った決め手 — D-cache が OFF であること**:
`system/mmu.c` の通り、このカーネルは DMA コヒーレンシのため **D-cache を意図的に OFF(SCTLR.C=0)**にしている。つまり**全データアクセスが RAM に直接届く**。結果:

> 各コアの読み書きはキャッシュを介さず即座に他コアから見える = **キャッシュコヒーレンシ問題が無い**。
> よって `volatile` 変数 + `dmb`/`dsb` バリアだけでコア間協調が成立し、**排他モニタ(spinlock / LDAXR-STLXR)は不要**(そもそも非キャッシュ・メモリでは排他モニタの挙動が実装依存で当てにならない)。

この事実が「コア1-3 を lock-free な計算ワーカーにする」設計を可能にした。

**レディリストについて**: 本ランタイムでは **レディリストをコア0専有のままにするのが SMP 最適**である(共有化は上記の非再入問題で逆効果)。並列性はワーカープールで獲得し、OS のスケジューラ/レディリストは無改造で安定性を保つ — という判断を明示的に行った。

## 3. 変更したファイル一覧(「どこを変えたか」)

新規 2ファイル + 既存 4ファイル。コア0の OS 動作は一切変えていない。

| ファイル | 行数 | 変更内容 |
|---|---:|---|
| **`include/smp.h`**(新規) | 41 | SMP 公開 API(`smp_init` / `smp_cores_online` / `smp_parallel_sum`) |
| **`system/smp.c`**(新規) | 161 | セカンダリコア起動・ワーカーループ・lock-free 並列ディスパッチ |
| `loader/boot.S` | +67 | セカンダリコアの起こし方(`_smp_start` トランポリン + `.data` ハンドオフ表) |
| `system/mmu.c` | +30 | `mmu_enable_secondary()`(各コアをコア0と同一の MMU/キャッシュ構成に) |
| `loader/main.c` | +8 | `kernel_main` から `smp_init()` を呼ぶ |
| `system/tcp_server.c` | +59 | ベンチマーク本体 + `/smp-bench` HTTP ルート |

以下、各変更の詳細。

### 3.1 `loader/boot.S` — セカンダリコアの起動経路

**変更前**: MPIDR≠0 のコア(1-3)は `1: wfe; b 1b` で**永久にパーク**(コメントに "we'll wake them later" とあるが未実装だった)。

**変更後**:
- セカンダリは `smp_release[core]` が設定されるまで待ち、設定されたら `_smp_start` へ収束する poll ループに変更。
- `_smp_start` トランポリンを新設: MPIDR からコアID取得 → **EL2→EL1 ドロップ(コア0と同じ手順)** → per-core スタック設定 → C の `smp_secondary_entry(core)` を呼ぶ。
- `.data` セクションに `smp_release[4]` / `smp_stacktop[4]` を配置(`.bss` だとコア0のクリア前にセカンダリが古い値を読む危険があるため、イメージロード時点で 0 になる `.data` に置いた)。
- 起動の取りこぼしを防ぐため、**2系統の起こし方**に両対応(§4)。

### 3.2 `system/smp.c`(新規) — ブリングアップ + 並列ディスパッチ

- `smp_init()`(コア0): per-core スタックを用意し、セカンダリを release してオンライン応答を**上限付き**で待つ(応答が無ければそのコアはオフライン扱い=ブートは止まらない)。
- `smp_secondary_entry(core)`(各セカンダリ): `exception_init()`(VBAR設定)→ `mmu_enable_secondary()`(MMU/キャッシュをコア0と同一に)→ オンライン宣言 → ワーカーループへ。
- `smp_worker_loop()`: `wfe` で低電力待機 → ジョブ投入(seq インクリメント+SEV)で起床 → 計算 → 結果書き戻し → done フラグ。
- `smp_parallel_sum(fn, n, ncores)`: `[0,n)` を ncores 個の連続範囲に分割。コア0は自分の範囲を**インライン実行**し、他コアにジョブを投入、全員の done を待って合計。**ワーカーが期限内に終わらなければコア0が肩代わり**(ハングしたコアが OS を巻き込まない安全弁)。

### 3.3 `system/mmu.c` — `mmu_enable_secondary()`

各セカンダリコアを **コア0と全く同じ MMU 構成**(コア0が構築済みの `l1_table` を共有、MMU on / I-cache on / **D-cache off**)にする。これにより4コアが同条件で実行され、**ベンチマークが公平**になる。ページテーブルは再構築せず TTBR0 を指すだけ。

### 3.4 `loader/main.c` — 呼び出し

`gic_init()` / `timer_init()` の後(MMUテーブル・例外ベクタ構築済みの地点)で `smp_init()` を1回呼ぶ。ブートログに `smp: cores online = 4` を出力。

### 3.5 `system/tcp_server.c` — ベンチマーク + HTTP ルート

- `bench_count_primes(lo,hi,core)`: 試し割りで `[lo,hi)` の素数を数える純整数計算(レジスタ内の mul/mod 中心 = メモリ帯域でなく CPU 律速。D-cache off でも素直にコア数でスケールする)。
- `now_ms()`: `CNTPCT_EL0`/`CNTFRQ_EL0` から ms 取得(高分解能)。
- `/smp-bench?n=<上限>`: 1コアで計測 → 全オンラインコアで計測 → 両者の素数の数(一致確認)・各時間・速度向上比(×100整数)を返す。

## 4. セカンダリコアをどう起こしたか(技術的核心)

Pi4 の標準ファームは、起動時にコア1-3 を **armstub のスピンテーブル**(物理 `0xe0`/`0xe8`/`0xf0`)で待機させ、そこに関数アドレスが書かれると SEV でそこへジャンプさせる。一方このカーネルの旧 `boot.S` は「全コアが kernel に来てパークする」前提だった。どちらの挙動か実機で確証が無かったため、**両系統に対応**した:

1. **スピンテーブル経路**: `smp_init()` が `0xe0/0xe8/0xf0` に `_smp_start` のアドレスを書き、SEV。
2. **boot.S パーク経路**: `smp_release[core]` に `_smp_start` を書き、SEV。boot.S の poll ループが拾う。

どちらの場所にいても **`_smp_start` に収束**する。実機結果 `cores_online = 4` は、この起こし方が正しく機能したことを示す(本機では実質スピンテーブル経路が効いている)。

**コヒーレンシ**: スピンテーブル mailbox(物理 0xe0 等)は `mmu.c` の L3 で **RW マップ済み**(`pa < _start` の領域)なので、コア0からの書き込みは fault しない。D-cache off ゆえセカンダリ(MMU off の armstub 内)からも即座に見える。

## 5. 安全設計(箱をハングさせないための作り込み)

- **コア起動失敗 → 縮退**: `smp_init` のオンライン待ちは上限付き(~数十ms/コア)。応答が無ければオフライン扱いでブート続行。最悪 `cores_online=1` でベンチはコア0が全部計算(正答・速度向上なしだがハングしない)。
- **ワーカー暴走 → 肩代わり**: `smp_parallel_sum` の done 待ちも上限付き。期限超過でコア0がその範囲を計算し直すので、**死んだワーカーが OS を巻き込まない**。
- **OS 無改造**: コア1-3 は実行中、共有 OS 状態(vheap/actor/net/video)に一切触れず、自分のジョブスロットと純計算のみ。コア0の協調スケジューラ・レディリストは元のまま。

---

## 6. 結果の分析 — なぜ 4倍でなく約3倍か

速度向上が安定して約 2.95–3.11倍に収まる理由は **SMP のオーバーヘッドではなく、ワークロードの負荷不均衡**:

- `smp_parallel_sum` は `[0,n)` を**等幅**の4範囲に分ける。しかし試し割りのコストは数 N に対して概ね √N に比例して**増える**。
- そのため コア0が担う `[0, n/4)` が最も軽く、コア3が担う `[3n/4, n)` が最も重い。全体の所要時間は**一番重い範囲**で決まる → 理論上限は約3倍に張り付く。
- 加えてコア0は OS のタイマー/ネットワーク割り込みも処理している。

つまり 4コアが正しく並列動作した上での**ワークロード特性**による頭打ちであり、SMP 機構は健全。等コストのワークロード(または範囲をストライド分割して負荷を均す)なら 4倍近くまで伸びる余地がある(将来拡張)。

## 7. 再現方法

```bash
# ビルド
cd compile && make pi4               # -> compile/kernel8.img

# デプロイ(SD コールドブート — セカンダリ起動はコールドブート必須)
cp compile/kernel8.img /Volumes/bootfs/kernel8.img && sync && diskutil eject /dev/diskN
#   ※ config.txt は温存(total_mem=2048)

# 実行(実機)
curl 'http://192.168.3.100/smp-bench?n=1000000'
```
ブート時シリアルに `smp: cores online = 4` が出る。
